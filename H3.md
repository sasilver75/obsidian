Lets you create a string index that's an alphanumeric code for any place on earth. Can understand the parents, neighbors, etc. Has become quite popular for visualization tools like [[kepler.gl]], [[CARTO]], etc.

A library form Uber that divides the Earth's surface into hexagonal cells at multiple resolutions.
Each cell has a unique ==64-bit integer ID==, often displayed as a [[Hexadecimal]] string like `882a100d2dfffff`.
- You can store this number as a BIGINT or as a TEXT - both work, but with some tradeoffs.
	- If you store as TEXT, you then use a regular [[B-Tree]] index on the text column. Works well, with equality lookups and group-bys being fast.
	- If you store as BIGINT (with h3-pg extension offering the native H#INDEX type, internally a bigint), you also use a B-Tree index, but get slightly more compact storage, and you get H3-aware SQL functions like `h3_cell_to_parent(...)`
- The bit layout of this integer is structured: the first few bits encode the resolution, and the next bits encode the ==base cell== (one of 122 coarse cells covering the earth), and the remaining bits encode the *path* down the hierarchy to a specific cell!
	- This is why the parent relationship is cheap to compute; it's just a bit mask operation, not a lookup.
	- But the shared prefix isn't quite as clean as a [[Geohash]], where you just truncate the string to go coarser.

The most obvious/common alternative would be [[Geohash]]es, which encode a lat/lon as a base-32 string (`9q5cs4...`), where a longer string is a more precise location. These are hierarchical, where truncating the string gives you the parent cell... Here, cells are rectangles, rather than hexagons. The big advantage is that it's just a string, so you can do prefix queries in any database without extensions (`WHERE geohash LIKE '9q5c%'`). Fast and simple but cells distort near the poles and have the 4/8 neighbor problem that hexagons avoid (see below).

Why hexagons?
- Each hexagon has ==exactly 6 equidistant neighbors==, unlike Squares (8, diagonals are further)... makes spatial smoothing and neighbor queries consistent. No gaps or overlays, and look better on maps.

Why use H3?
- Hierarchical: ==Every resolution-8 cell contains exactly 7 resolution-9 cells==, which contain exactly 7 resolution-10 cells, etc. So you can aggregate up and down the hierarchy cheaply.
- Universal: Any lat/lon -> H3 cell index is a pure function, ~O(1). No polygon-in-polygon lookup needed.
- PostGIS integration: Use the `h3-pg` extension to add H3 functions to Postgres.

Typical use vs specific Geometries: Typically you would use an H3 column + B-Tree for the "paint the city" aggregation views, and then a geometry column + a [[GiST]] index for "what's near this specific point" detail queries. They're complementary!


Resolutions:
```
Resolution 6  ≈ 36 km²  per cell  — city-scale (LA has ~60 cells)
Resolution 7  ≈ 5.2 km² per cell  — large neighborhood
Resolution 8  ≈ 0.7 km² per cell  — a few city blocks      ← analysis sweet spot
Resolution 9  ≈ 0.1 km² per cell  — one block face
Resolution 10 ≈ 0.015 km² per cell — half a block          ← storage resolution
Resolution 12 ≈ 0.0003 km² per cell — building footprint scale
```
So higher resolution cells are smaller cells.
- A TEXT column storing `8c2a100d2d6ffff` is 15 characters regardless of resolution.
- The number of *distinct* H3 values in your index grows if you have higher resolution, but the *per-row storage cost is identical!* If you have 10M rows, you have 10M H3 values either way.
	- The actual cost is in:
		- Index size and B tree performance (At resolution 12, you approach one-row-per-cell, because incidents rarely repreat in the same 15m hex; the index degenerates towards a unique index, which offers less benefit for GROUP BY queries.)
		- Aggregation queries become slower at coarser resolutions (If you query at resolution 8 from resolution 12 data, your `h3_cell_to_parent()` has to traverse 4 levels instead of 2! Most importantly, Postgres can't use the index efficiently for the GROUP BY: it has to compute the parent for every single row before grouping.)
		- Precision-washing argument: Resolution 12 (~15m) might be finer than the actual geocoding accuracy of most datasets -- it implies that you know which building something happened at, where you actually only know *which block*. The false preciison can mislead analysis.


At ingest time, for every record with a lat/lon:
```python
import h3

lat, lon = 34.0522, -118.2437
h3_index_r10 = h3.latlng_to_cell(lat, lon, resolution=10)
# → '8a2a100d2d6ffff'
```

Store `h3_index_r10` as a column. Then aggregate at query time:
```python
# Get the parent cell at resolution 8
h3_r8 = h3.cell_to_parent(h3_index_r10, 8)
# → '882a100d2dfffff'
```

In SQL with h3-pg:
```sql
-- Aggregate 311 calls to resolution-8 hexagons
SELECT
  h3_cell_to_parent(h3_r10, 8) AS h3_r8,
  COUNT(*) AS call_count
FROM service_requests
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY h3_r8;
```

==For frontend use==: Your API returns a list of `{h3_index, count}` pairs, and the frontend uses the H3 JS library to convert each index back to a hexagon polygon for rendering.

While [[H3]] was designed by Uber for ridesharing in cities, it's been taken and used in many contexts where it doesn't make sense.
