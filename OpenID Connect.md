A standard login/authentication layer built on top of [[OAuth|OAuth 2.0]]. 
- Can also be sued to implement [[Single Sign-On]] (SSO).

While OAuth 2.0 mainly answers: "Can this app access this API/resource?"
OIDC answers "Who is the user that just signed in?"

It does this by adding an `id_token` to the OAuth flow. This `id_token` is usually a signed [[JSON Web Token|JWT]] containing identity claims such as:
```json
{
    // Issuer: the identity provider that created/signed this token
    "iss": "https://accounts.example.com",

    // Subject: the stable unique user ID within that issuer
    "sub": "user_123",

    // Audience: the intended recipient of this token, usually your client/app ID
    "aud": "my-client-app",

    // Expiration time: Unix timestamp after which the token is no longer valid
    "exp": 1760000000,

    // User email claim; useful for display, but usually not the primary user ID
    "email": "alice@example.com"
  }
```


# Typical OIDC flow
1. User clicks "Sign in"
2. App redirects user to identity provider
3. User authenticates
4. Identity provider redirects back with an authorization code
5. App exchanges the code for tokens
6. App validates id_token
7. App creates its own [[Session]] for the user

# Example

Imagine that we have a web app called ==Acme Notes== that lets users sign in with Google.

Actors:
- User: Alice
- Browser: Alice's browser
- Client App: `http://app.acme-notes.com`
- Identity Provider: `https://accounts.google.com`
- App Backend: Acme's server

Before any user logs in, ==Acme registers its app with Google== and, and gets back:
```
client_id: acme-notes-client-id
client_secret: only used by backend/server apps
redirect_uri: https://app.acme-notes.com/auth/callback
```

For the log-in flow:

1. Alice clicks "Sign in with Google"
The app creates temporary security values:
```
state: prevents CSRF/login response mixups
nonce: binds the returned ID token to this login attempt
code_verifier/code_challenge: PKCE protection
```

2. Acme redirects Alice's browser to Google
```
GET https://accounts.google.com/o/oauth2/v2/auth
    ?response_type=code
    &client_id=acme-notes-client-id
    &redirect_uri=https://app.acme-notes.com/auth/callback
    &scope=openid email profile
    &state=random_state_123
    &nonce=random_nonce_456
    &code_challenge=hashed_code_verifier
    &code_challenge_method=S256
```
The important part is: `scope=openid email profile`
- `openid` makes this an [[OpenID Connect|OIDC]] login flow, rather than just plain [[OAuth]]

3. Google authenticates Alice
Google might ask for:
- email/password
- MFA
- passkey
- account selection
- consent
Here, Acme does not see Alice's Google password; Authentication happens at Google.

3. Google redirects the user back to Acme's callback URL, delivering a `code` + `state`
```
GET https://app.acme-notes.com/auth/callback
    ?code=temporary_authorization_code
    &state=random_state_123
```
- This `state` was generated by Acme itself before redirecting to Google, meaning: "This callback belongs to the login attempt I started earlier." In a way, it's Acme's correlation/[[Cross-Site Request Forgery|CSRF]] token for this auth attempt.
- This `code` is short-lived; it's not the final token. It's generated by Google, and means roughly "Google says this browser completed authentication/consent for this client, and this temporary grant can be redeemed for tokens." It is not the final proof of identity, it is a redeemable artifact.

5. Acme checks `state`
Acme verifies: 
- Returned state == state it originally generated
	- If not, reject the login

6. Acme exchanges the `code` for tokens
The backend calls Google directly:
```
POST https://oauth2.googleapis.com/token

With something like
{
    "grant_type": "authorization_code",
    "code": "temporary_authorization_code",
    "redirect_uri": "https://app.acme-notes.com/auth/callback",
    "client_id": "acme-notes-client-id",
    "client_secret": "server-side-secret-if-app-has-one",
    "code_verifier": "original_code_verifier"
}
```

7. Google returns tokens
```
{
    "id_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6Ij...JWT...",
    "access_token": "token-for-calling-google-apis",
    "expires_in": 3600,
    "refresh_token": "optional-long-lived-token"
}
```
For login, the key token here is the `id_token`. This is a [[JSON Web Token|JWT]], and is the identity proof, from Acme's perspective.
- The *access token* may or may not be a JWT. That depends on the provider. Some access tokens are JWTs; some are opaque strings. 
Q: What's the access token vs the id token?
- The `id_token` is for the client application, it proves who authenticated.
- The `access_token` is for the API/resource server, ti grants access to something. It's an OAuth concept; If Acme only needs login, it may only need the `id_token`, and not even care about the `access_token`, but if Acme also wants to read Alice's Google Calendar (a la usual [[OAuth]] flow), it can use the `access_token` with an appropriate scope.


8. Acme does not just decode it, it verifies:
```
signature: signed by Google
iss: token came from expected issuer
aud: token is meant for Acme's client_id
exp: token is not expired
nonce: matches the login attempt
```
The decoded ID token might contain:
```
{
    "iss": "https://accounts.google.com",       // issuer. must be expected issuer
    "sub": "109837465",                         // subject. stable Google user ID
    "aud": "acme-client-id",                    // audience. must be Acme's client_id
    "exp": 1760000000,                          // expiry. must not be expired
    "iat": 1759996400,                          // issued at. issued-at sanity check
    "nonce": "random_nonce_456",                // must match original nonce
    "email": "alice@gmail.com",
    "email_verified": true
}
```
Acme verifies the ID token using the provider's [[Asymmetric Key Encryption|Public Key]]s
1. Decode JWT header, read `kid`
2. Fetch google's [[JSON Web Key Set]]/public keys
3. Pick matching public key
4. Verify the JWT [[Signature]]
5. Validate the claims
- 

 9. Acme maps the external identity to a local user
Usually, this key is:
- issuer + subject
For example:
`google:user 109837465 -> Local Acme user id 42`
Important: Use `sub` s the durable identity, not email (which can change)

10. Acme creates its own app [[Session]]
After OIDC proves who Alice is, Acme usually creates a normal session cookie:
```
Set-Cookie: acme_session=abc123; HttpOnly; Secure; SameSite
```
From then on, Alice uses Acme like a normal logged-in user:
```
Browser -> Acme: session cookie
Acme -> database: load local user 42
```
Key Design Point:
- OIDC is usually used only at login time

Q: Why does it have this back-and-forth thing? I totally understand how we have the initial redirect from the application to Google, and that Google can authenticate to Alice and then redirect back to Acme, but why doesn't Google then just return the actual token as part of the redirect? Why does Acme have to do another round trip to exchange that return code for a token?
- ((Oops, this is a pretty silly question.))

A: Because the browser redirect is a leaky, user-controlled channel... We want to let Google deliver the real tokens over a direct server-to-server request after proving that the same legitimate app that started the login after finishing it.

Q: I know that OIDC builds on OAuth, and that OAuth is used for Authz while OIDC is used for Authn, but I'm just surprised that the Authn thing builds on the Authz thing, given that I usually think that authentication is a prerequisite for authorization. Can you help put me straight?

A: The trick is that OIDC builds on OAuth because it already had the useful machinery: (redirect to provider, client registration, authorization endpoint, token endpoint, scopes, tokens, refresh, JWKS/discovery patterns). OIDC reused that machinery and added a standard identity result.
- A better mental model:
	- OAuth 2.0: Framework for getting tokens from an authorized server
	- OIDC: Profile of OAuth 2.0 where one token proves user authentication

OAuth 2.0 came out for use cases like "Allow Acme to read my Google Calendar." To do this, Google usually has to authenticate Alice first, but OAuth doesn't standardize the authentication result. It basically says "Google figures out who Alice is however it wants, asks for consent if needed, and gives us back an access token."
- So user authentication *does happen* inside the OAuth flow, but OAuth doesn't expose some sort of standardized "Alice logged in" proof to the client.

OIDC says: "Since the trusted provider themselves already authenticated Alice, also return a standardized `id_token` that tells Acme who Alice is!"
1. Alice clicks "Sign in with Google" in Acme
2. Acme redirects Alice to Google with: `client_id, redirect_uri, scope=openid, email prfile, state, nonce, PKCE challenge`
3. Alice authenticates to Google, however Google likes
4. Google decides Acme is allowed to receive the requested identity claims
5. Google redirects the browser back to Acme: `/auth/callback?code=AUTH_CODE&state=STATE`
6. Acme verifies state
7. Acme sends AUTH_CODE to Google's token endpoint: `codce, client_id, client_secret or PKCE verifier, redirect_uri`
8. Google returns tokens: `id_token, maybe access_token, maybe refresh_token`
9. Acme validates `id_token`
10. Acme maps the Google-returned identity to a local Acme user
11. Acme creates and stores its own session cookie (SessionID hashed, stored in Redis, Set-Cookie with the real SessionID to the client)
12. Later, Acme authenticates requests from Alice using her supplied session cookie, looking up her existing session in (e.g.) Redis via hash of the user-supplied session ID. It then authorizes actions based on Acme's understanding of Alice's roles, etc.

If we wanted to have much more coarse perspective, it's something like:
1. Start Login: Alice chooses to Login with Google and gets redirected
2. Provider (Google) Authentication and Consent: Alice authenticates to Google
3. Front-Channel Callback: Google redirects browser to Acme + Acme verifies state
4. Back-Channel  Token Exchange: Acme redeems the temporary `code` with Google for a real token
5. External Identity Verification: Acme validates Google-returned `id_token` JWT, maps to a local Acme user



Q: Okay, when you say "Acme maps the Google-returned identity to a local Acme user," what does that look like, again?

A: Recall that Google has returned to Acme something like:

```
{
    "id_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6Ij...JWT...",
    "access_token": "token-for-calling-google-apis",
    "expires_in": 3600,
    "refresh_token": "optional-long-lived-token"
}

Where id_token, decoded, looks something like:
{
    "iss": "https://accounts.google.com",       // issuer. must be expected issuer
    "sub": "109837465",                         // subject. stable Google user ID
    "aud": "acme-client-id",                    // audience. must be Acme's client_id
    "exp": 1760000000,                          // expiry. must not be expired
    "iat": 1759996400,                          // issued at. issued-at sanity check
    "nonce": "random_nonce_456",                // must match original nonce
    "email": "alice@gmail.com",
    "email_verified": true
}
```

From this, Acme looks at the issuer (`iss`) and subject (`sub`). 

We use the two of these to look up the Acme user identity in some sort of Acme auth service:
- We might have a Postgres table: `external_identities`:

```
Postgres table: external_identities
    issuer
    subject
    user_id
    provider
    created_at
    last_login_at

Uniqueness constraint: (issuer, subject)
```

Then at login:
1. Acme validates Google's `id_token`
2. Acme extracts `iss` + `sub`
3. Acme queries Postgres: `SELECT user_id FROM external_identities WHERE issuer=... AND subject=...`
4. Acme creates a session for that user_id

You can cache this information if needed.

Q: Why doesn't Google just include the real OIDC token in the redirect back to Acme? Why make Acme exchange an authorization code for the token in another round trip?

A: 
- Because redirect URLs are browser-mediated, and can leak through history, logs, referer headers, frontend code, extensions, analytics, or copied links. 
- The temporary authorization code (on the other hand) is safer to send through the redirect because it is short-lived, single-use, and useless without [[Proof Key for Code Exchange|PKCE]] or client-secret proof. 
- Acme then redeems it over a direct back-channel to Google, where Google can verify the legitimate client before issuing real tokens.
